Mace stands for **M**ain **A**pplication **C**onfiguration **E**ngine. It's a language that's designed for type safe configuration.
Instead of relying on Editors to decide how to enforce schemas it's built in! External data can be validated by the CLI itself! 
The built in LSP server allows the developer to warned about what data will be inserted into the app! 


<Aside title="Need reuse values that have already been writen in the config">

Just use `$self`! 

```mace 
{
  first_name: 'Shelton',
  last_name: 'Louis', 
  full_name: "$($self.first_name)  $($self.last_name)"

}
```
</Aside>

<Aside title="Need to write schemas for other files to use?">

Make a mace file using the output directive to `schema` to indicate that you're making a schema instead of writing data.

```mace
[output=schema]
{
  name: string, 
  version: string,
  dependencies: array<{
    name: string,
    version: string,
    scope?: string,
  }>,
}
```

You can then import the schema you have made and make sure that the config you're writing conforms to it! 

```mace
[schema_file='./package-schema.mace']
  {
    name: '@code-fixer-23/cn-efs',
    version: "3.0.1",
    dependencies: [
      {
        name: 'clsx',
        version: '^2.0.0',
      }
    ],
  }

```

</Aside>

<Aside title="Need to make sure that data that is supposed to be used for the config exists?">

Use the parse file directive to link to a file!

```mace

[parse_file='./github-context.mace']
{
name: 'CI',
on: 'push',
jobs:{
  prod_check:{
    if: $github.ref == 'refs/heads/main',
    runs_on: 'ubuntu-latest',
    steps: [
      { run: "echo 'Deploying to production server on branch $GITHUB_REF'",}
    
    ]
  
  }

}

}
```
</Aside>

## What this language about? 

This one is a structural, strongly typed config language that's designed to allow developers to write config's that are modern and understandable.
Developers write config data using records that all come together to make a big record for people to use! 
The config data is typed using schemas! Which are record shaped types enforce the structure of records. 
It gives developers, strings, ints hex ints, floats, hex floats and booleans as data values.
As data structures it gives developers arrays and records. 

If a developers want's to specify that a value must be one or more type's the `variant` type exists; 
If a developer wants to specify that two or more schemas can be combined to make a new one the `union` type exists;
If a developer wants to indicate that a value must be a literal set of values they have the choice type.

## Before using this Language

When it comes to using Mace I need you to know that this language was created so that other languages could easily use it.
For other tools to make marshallers for it without any friction. To make sure that this language expreses the ideas most developers need! 

<Aside title="Here are the restrctions and reasons!" type="caution">

| Restriction                                                                    | DX Reason                                                                                                                                                                                            |
| :----------------------------------------------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kebab-case identifiers cannot be used                                          | The `-` character is the subtraction operator. Rejecting kebab case prevents expressions such as `foo-bar` from being ambiguous and keeps parsing independent of whitespace.                         |
| Field names cannot be string literals                                          | Mace uses one consistent field-access model based on identifiers and the `.` operator. This avoids forcing developers and tooling to support both `record.field` and `record["field-name"]`.         |
| Identifiers cannot contain arbitrary punctuation                               | Restricting identifiers to Unicode letters, Unicode digits, and `_` makes autocomplete, rename operations, member access, diagnostics, and conversion to other formats more predictable.             |
| Identifiers must begin with a Unicode letter                                   | This prevents identifiers from being confused with numeric literals, operators, directives, or other syntax.                                                                                         |
| Identifiers are case-sensitive                                                 | Case-sensitive names avoid implicit name transformations and ensure that source text, imports, schemas, and generated output refer to exactly the same symbol.                                       |
| Reserved keywords cannot be used as identifiers                                | Reserving language keywords prevents declarations from becoming ambiguous as Mace adds syntax and tooling features.                                                                                  |
| Mace has no `null` runtime value                                               | Mace models absence through optional fields rather than requiring every consumer to handle both omission and `null`. This produces clearer schemas and fewer defensive checks.                       |
| Optional fields are omitted rather than assigned `null`                        | A field is either present with a valid value or absent. Developers do not have to distinguish between missing, explicitly null, and incorrectly typed values.                                        |
| Script variables require explicit type annotations                             | Explicit declaration types document intent, improve diagnostics and autocomplete, and keep imported and injectable values predictable. Mace may still determine the types of expressions internally. |
| Every variable must have an initializer                                        | Variables always enter the value environment with a validated value, preventing partially initialized configuration state.                                                                           |
| Variables are immutable                                                        | Configuration values cannot silently change during evaluation. A declaration always means the same thing everywhere it is referenced.                                                                |
| A Mace document must contain exactly one output block                          | Every file has one explicit export surface. Tooling never has to guess which value represents the document result.                                                                                   |
| The output block must be a record                                              | Named fields provide a stable export surface for imports, schemas, documentation, autocomplete, and `$self`. Arrays remain available as field values.                                                |
| An output block may be empty, but it cannot be omitted                         | `{}` explicitly communicates that the document exports an empty record. An absent output block is more likely to indicate an incomplete file or authoring error.                                     |
| A document may contain at most one script block                                | A single declaration area gives developers and tools one predictable place to find imports, types, schemas, documentation, and variables.                                                            |
| The script block must appear before the output block                           | Declarations and imports are established before output evaluation. Readers and tools can process the document from top to bottom without forward execution dependencies.                             |
| Imports must appear before other script declarations                           | Developers can see external dependencies immediately, while processors can resolve the complete import environment before validating local declarations.                                             |
| Wildcard imports are not allowed                                               | Named imports make dependencies visible, improve autocomplete, prevent accidental name collisions, and avoid silently expanding a file’s public API.                                                 |
| Only names surfaced through an imported file’s output block are importable     | Script declarations remain private by default. A file’s output block acts as an explicit public interface rather than exposing implementation details.                                               |
| Imported names cannot be shadowed by local declarations                        | A name always resolves to one predictable declaration. Refactoring an import cannot silently change the meaning of local expressions.                                                                |
| All declarations share one namespace                                           | A type, schema, choice, variable, or imported symbol cannot reuse the same name. This eliminates context-dependent name resolution and makes navigation clearer.                                     |
| Duplicate declarations, fields, and directives are rejected                    | Mace never relies on “first value wins” or “last value wins” behavior. Duplicate names are surfaced as mistakes instead of silently changing configuration.                                          |
| Schemas are closed by default                                                  | Records validated against a schema cannot contain unknown fields. Typos and unsupported configuration options are reported instead of being silently ignored.                                        |
| Required schema fields cannot be omitted                                       | Consumers can rely on required data being present after validation rather than repeating existence checks.                                                                                           |
| Optional schema fields must still satisfy their declared type when present     | Optionality controls presence, not type safety. A present optional field receives the same validation as a required field.                                                                           |
| Arrays must be homogeneous unless their element type explicitly uses a variant | Array consumers receive a predictable element type. Mixed arrays must be intentionally modeled rather than occurring accidentally.                                                                   |
| Array access requires an integer literal index                                 | Literal indexes allow Mace to validate the access ahead of evaluation and provide precise out-of-range diagnostics.                                                                                  |
| Out-of-range array access is rejected                                          | Configuration evaluation fails at the invalid access instead of producing an undefined or nullable value.                                                                                            |
| `$self` is available only during data-output construction                      | `$self` has one clear meaning: the output record currently being produced. It cannot expose unrelated script or host state.                                                                          |
| `$self` can access only fields that were already evaluated                     | Output dependencies remain visible in source order. Forward references and circular field dependencies cannot occur implicitly.                                                                      |
| Schema output is type-only                                                     | Schema files describe shapes rather than runtime values. Variables and value expressions cannot accidentally leak into a schema export.                                                              |
| Schema output cannot be combined with data-validation directives               | A block either produces schema metadata or runtime data. Keeping the modes separate gives tooling and importers one unambiguous result kind.                                                         |
| Type declarations are not runtime values                                       | Schemas, aliases, and choices cannot be accidentally emitted, interpolated, or used as normal data. The schema plane and value plane remain separate.                                                |
| String interpolation accepts runtime values only                               | Interpolation produces text from evaluated data. Types and schemas cannot be unintentionally converted into unstable implementation-specific strings.                                                |
| Single-line strings cannot span multiple lines                                 | Accidental missing quotes are diagnosed close to their source. Multiline content must use an explicit block-string form.                                                                             |
| Mace does not perform implicit numeric conversions                             | An `int`, `float`, `hex_int`, and `hex_float` remain distinct unless the language explicitly defines an operation between them. This prevents silent precision or representation changes.            |
| Decimal and hexadecimal numeric families cannot be mixed implicitly            | Hexadecimal values preserve their intended representation and operator semantics rather than quietly becoming decimal numbers.                                                                       |
| Fusion members must be schemas                                                 | Fusion represents composition of record contracts, not a general-purpose union of unrelated scalar values. Developers can predict that its result is one combined schema shape.                      |
| Conflicting fusion fields are rejected                                         | Schema composition cannot silently choose one field definition over another. Conflicts must be resolved deliberately by the author.                                                                  |
| A variant value must match exactly one member                                  | A value that matches multiple alternatives is ambiguous. Requiring one exact match makes validation results and autocomplete deterministic.                                                          |
| Structural merge operates only on compatible records or arrays                 | Merge remains a data-composition operation rather than becoming an implicit conversion mechanism for unrelated values.                                                                               |
| Records and arrays cannot be merged directly with each other                   | Rejecting cross-kind merges prevents surprising result shapes and makes the result type knowable before evaluation.                                                                                  |
| Expressions cannot perform arbitrary code execution                            | Configuration evaluation cannot launch processes, call host-language functions, or execute strings as code. Opening a configuration file does not execute application behavior.                      |
| Expressions cannot perform network or arbitrary filesystem operations          | Evaluation depends only on declared source files, imports, and explicit runtime input. Builds remain reproducible and safe to inspect.                                                               |
| Import and schema-file paths must remain inside the project root               | A configuration file cannot escape its project and read unrelated or sensitive files through path traversal.                                                                                         |
| Circular imports are rejected                                                  | Import resolution always terminates, and exported values never depend on an incomplete declaration cycle.                                                                                            |
| Runtime input must satisfy a selected schema before output evaluation          | External data is validated at the trust boundary. Output expressions never receive malformed or partially valid input.                                                                               |
| Runtime input is treated only as data                                          | Input strings cannot create imports, directives, declarations, file reads, or executable expressions.                                                                                                |
| Source files must use UTF-8                                                    | One required encoding eliminates platform-dependent decoding behavior while still supporting international strings and identifiers.                                                                  |
| Unicode text is not automatically normalized                                   | Mace preserves the exact code points written by the developer instead of silently modifying identifiers or string values.                                                                            |
| Documentation cannot affect evaluation                                         | Documentation remains safe metadata. Adding or changing documentation cannot alter configuration results.                                                                                            |
| The processor emits no partial result after an error                           | Consumers receive either a completely valid result or a diagnostic. They never have to determine which portion of a failed configuration is trustworthy.                                             |

</Aside>
